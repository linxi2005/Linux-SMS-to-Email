# Linux 4G 短信转发系统（Python）开发需求整理版

## 一、项目定位

开发一个运行在 Linux 系统上的 Python 项目，用于通过 USB 连接的 4G 模块实时接收短信，并根据用户配置自动转发到电子邮箱。

本整理版需求以最新补充要求为准，已移除重复和冲突项，统一采用以下原则：

- 以移远 EC20 系列模块为当前唯一重点支持对象。
- 以 Linux 小盒子设备为主要部署场景。
- 优先兼容 ARMv7、Ubuntu 20 等较低配置和较旧系统环境。
- 优先兼容较低 Python 版本，避免依赖版本过高导致安装困难。
- 运行时业务数据唯一存储到 SQLite。
- 短信接收流程以 `+CMTI` 通知机制为准，不使用 `+CMT` 直推解析作为主流程。

## 六、多模块支持原则

系统必须支持多个 EC20 模块同时在线工作。

例如：

```text
/dev/ttyUSB2
/dev/ttyUSB5
/dev/ttyUSB8
```

要求：

- 每个模块独立初始化
- 每个模块独立监听
- 每个模块独立同步短信
- 每个模块独立状态管理
- 任意模块异常不得影响其他模块
- 任意模块掉线后恢复不应影响其他模块正常接收短信

### 1. 模块配置持久化

所有手动添加的模块信息自动保存到：

```text
config/modem_config.json
```

建议格式：

```json
{
  "modems": [
    {
      "port": "/dev/ttyUSB2",
      "remark": "EC20-1"
    },
    {
      "port": "/dev/ttyUSB6",
      "remark": "EC20-2"
    }
  ]
}
```

要求：

- 程序启动时自动读取 `config/modem_config.json`
- 自动恢复所有手动添加的模块
- 如果文件不存在，则自动创建
- 新增模块后立即保存
- 修改备注后立即保存
- 删除模块后立即保存
- 不需要等待程序退出才写回配置

### 2. 模块基础信息

每个模块至少维护以下信息：

- 模块 ID
- 串口号
- 用户备注
- 型号

- IMEI
- ICCID
- IMSI
- SIM 状态
- 运营商
- 网络类型
- 信号强度

这些信息用于后台展示和日志记录。

### 1. 串口独占

要求：

- 不允许多个本程序实例同时操作同一串口

- 设置短信存储位置
- 设置短信到达通知方式
- 读取型号
- 读取固件版本
- 读取 IMEI
- 读取 ICCID
- 读取 IMSI

- 读取信号强度
- 读取运营商
- 读取网络注册状态
- 读取网络制式

这些信息应保存到数据库并用于模块状态展示。

## 八、模块初始化 AT 流程

### 1. 核心初始化方式

短信初始化监听流程统一采用以下指令逻辑：

```text
ATE0
AT+CMGF=1
AT+CPMS="MT","MT","MT"
AT+CNMI=2,1,0,0,0
```

各指令含义：

- `ATE0`：关闭 AT 回显，方便程序解析。
- `AT+CMGF=1`：设置短信文本模式。
- `AT+CPMS="MT","MT","MT"`：读写存储统一使用 MT（模块 + SIM）。
- `AT+CNMI=2,1,0,0,0`：收到短信后保存到模块，并输出 `+CMTI` 新短信索引通知。

### 2. 初始化补充信息读取

除了上述短信相关初始化，还必须读取以下模块信息：

```text
ATI
AT+CGMM
AT+CGMR
AT+CGSN
AT+CPIN?
AT+CSQ
AT+COPS?
AT+CREG?
AT+CGREG?
AT+QCCID
AT+CIMI
```

说明：

- 若个别指令在某些 EC20 固件版本返回格式不同，程序必须做兼容解析。
- 若某个附加信息读取失败，不能导致模块整体初始化失败。
- 关键初始化失败应写入日志并更新模块状态。

### 3. 初始化结果要求

初始化完成后，模块状态页应可展示：

- 模块是否在线
- 型号
- 固件版本
- IMEI
- ICCID
- IMSI
- SIM 状态
- 运营商
- 网络类型
- 信号值
- 最近初始化时间

## 九、短信接收机制

### 1. 监听方式

程序后台持续监听串口返回内容。

主监听目标为：

```text
+CMTI:
```

例如：

```text
+CMTI: "ME",8
```

或：

```text
+CMTI: "SM",8
```

### 2. 新短信处理流程

程序收到 `+CMTI` 后必须：

1. 自动解析存储位置，如 `ME`、`SM`、`MT`
2. 自动解析短信索引，如 `8`
3. 记录通知日志
4. 自动执行：

```text
AT+CMGR=8
```

5. 解析短信内容
6. 写入 SQLite
7. 尝试发送邮件
8. 根据处理结果更新短信状态

### 3. `+CMGR` 解析内容

至少解析以下字段：

- 发件号码
- 接收时间
- 短信正文
- 已读/未读状态
- 短信索引
- 存储位置
- 编码类型
- 来源模块

### 4. 编码处理要求

程序必须自动识别短信编码：

- 普通 ASCII / GSM 文本直接使用
- 如果正文是 UCS2 十六进制字符串，则自动按 UTF-16BE 解码
- 必须兼容英文短信
- 必须正确显示中文短信

### 5. 已读处理

程序成功读取短信后，应自动执行相应处理将短信标记为已读。

要求：

- 读取完成后更新模块内短信状态为已读
- 数据库中也要保存已读状态
- 若标记已读失败，不能影响短信已保存和邮件转发结果，但必须记录日志

## 十、短信接收 Bug 修复要求

### Bug #6：短信收不到（模块发送 CMTI 但程序不识别）

#### 1. 症状

- 发短信到模块，模块指示灯正常
- 程序日志无任何反应
- SQLite 无新记录

#### 2. 根因

`_listen_loop` 只匹配了 `+CMT:`，完全没有处理 `+CMTI:`。

而绝大多数 4G 模块默认发送：

```text
+CMTI: "ME",8
```

程序直接跳过，导致短信无法入库。

#### 3. 修复要求

- 监听线程必须优先支持 `+CMTI:`
- `CNMI` 使用 `2,1,0,0,0`
- 新短信读取必须走 `CMTI -> CMGR(index)` 流程
- 不再以 `+CMT` 直推模式作为主方案

#### 4. 经验约束

需要明确认识：

- `CNMI=2,2` 常见表现是直接输出 `+CMT`，短信不一定存储
- `CNMI=2,1` 常见表现是输出 `+CMTI`，短信先存储再通知

本项目统一采用后者。

## 十一、短信同步机制

除了实时监听短信外，系统还必须支持定时同步模块内短信。

### 1. 默认同步方式

- 默认每 30 秒同步一次
- 同步周期可配置
- 不允许过于频繁，避免低配设备负担过高

### 2. 同步内容

同步命令可采用：

```text
AT+CMGL="ALL"
```

要求：

- 可读取模块内已读和未读短信
- 程序自动判断本地是否已存在
- 程序自动判断是否已成功转发
- 仅对新增短信或状态变化短信进行处理

### 3. 去重机制

短信去重统一采用哈希值。

推荐规则：

```text
SHA256(号码 + 时间 + 短信内容 + IMEI)
```

要求：

- 数据库必须建立唯一索引
- 防止程序重启后重复入库
- 防止同步和实时监听重复写入

### 4. 转发补偿

如果：

- 模块中存在短信
- 本地已存在短信记录
- 但邮件未成功转发

则：

- 程序应重新执行邮件发送
- 成功后更新转发状态和时间

- 首次启动自动创建数据库和表结构

- 长期运行不能因短信数量增长而明显增加内存占用
- 写入必须使用事务，避免异常导致数据损坏

## 二十七、后台运行要求

### 1. 长期稳定运行

要求：

- 后台线程持续运行

- 任意模块异常自动隔离
- 模块重启后自动恢复监听
- 程序重启后自动恢复模块配置并重新初始化

- 重连不要太频繁
- 使用退避机制
- 重连期间避免阻塞其他模块

## 二十八、性能优化要求

面向 ARMv7 和 Ubuntu 20 小盒子，必须重点优化：

- 所有数据库查询建立索引
- 所有列表采用分页
- 不允许一次性读取全部短信到内存
- Web 页面只查询当前页需要的数据
- 后台同步只处理新增或变化记录
- 日志量可控
- 线程数量受控
- 定时任务频率可配置
- 避免使用重量级依赖
- 避免使用编译困难或 ARMv7 下安装复杂的依赖

依赖版本策略：

- 优先选择纯 Python 或常见 manylinux 可安装包
- 尽量避免需要复杂本地编译的依赖
- requirements.txt 中固定使用稳定、兼容、可安装的版本范围

## 三十一、最终结论

本项目最终采用以下统一口径：

- 当前只重点支持移远 EC20 系列。
- 多模块支持只围绕多个 EC20 实现。
- 短信接收以 `AT+CNMI=2,1,0,0,0` + `+CMTI` 通知机制为主。
- 收到 `+CMTI` 后通过 `AT+CMGR=index` 读取短信。
- 程序启动时必须完整初始化模块并独占锁定串口。
- 业务数据唯一存储到 SQLite。
- 模块配置持久化到 `config/modem_config.json`。
- 重点兼容 ARMv7、Ubuntu 20 和较低 Python 版本环境。
- 保活和重连必须温和，避免过于频繁影响收短信。
- Web 后台需提供模块管理、短信历史、手动读模块短信、手动重发邮件、AT 调试、数据库维护等完整能力。
