# Linux 4G短信转发系统（Python）完整开发需求

请开发一个**运行在 Linux 系统上的 Python 项目**，用于通过 USB 连接的 4G 模块实时接收短信，并根据用户配置自动转发到电子邮箱。

整个项目要求**工程化开发**，代码规范、模块化、可维护，并提供现代化、美观的 Web UI。

# 十八、短信同步机制（新增）

除了实时监听短信外，系统还必须支持定时同步模块内短信。

默认：

每30秒同步一次（可配置）。

同步流程：

1、读取模块短信列表。

例如：

```
AT+CMGL="ALL"
```

或者根据不同品牌自动选择最合适的AT命令。

读取所有短信后：

程序自动判断：

* 本地是否已经存在
* 模块短信是否已读
* 是否已经成功转发

如果：

模块中存在

本地不存在

则：

立即保存至本地。

如果：

本地存在

但是：

未完成邮件转发

则：

重新执行邮件发送。

邮件发送成功后：

自动执行：

```
AT+CMGR=index

AT+CMGD（根据配置）

或

标记短信为已读
```

要求：

已经成功转发的短信：

必须自动标记为已读。

未成功转发：

保持未读。

程序重启后：

自动同步模块所有短信。

不能因为程序离线导致短信遗漏。

同步时必须自动去重。

推荐使用：

号码 + 时间 + 内容

生成唯一Hash进行去重。

# 二十三、数据存储架构（重要）

为了保证系统长期稳定运行，提高查询效率，降低内存占用，项目**必须使用 SQLite 作为唯一的运行时数据库**，不再使用 `sms.json` 存储短信数据。

SQLite 数据库默认保存为：

```text
data/sms.db
```

SQLite 应作为系统所有业务数据的唯一数据源。

配置文件（如 `config.json`、`mail.json`）仍然使用 JSON 格式保存。

---

## 数据库要求

程序首次启动时应自动创建数据库及数据表，无需用户手动初始化。

所有数据库操作均需封装到独立模块，例如：

```text
database/
├── database.py
├── models.py
├── sms.py
├── modem.py
├── mail.py
```

数据库操作统一封装，不允许在业务代码中直接拼接 SQL。

推荐使用：

* SQLAlchemy（推荐）
* Peewee（轻量级 ORM）
* 或 SQLite3 + 参数化 SQL（必须做好封装）

---

## 开发原则（重要）

本项目必须采用"数据库驱动"架构，而非"JSON 文件驱动"架构。

SQLite 是系统唯一的数据源。

JSON 仅用于：

* 系统配置
* 邮件配置
* 配置导入/导出

所有短信、模块信息、邮件发送记录、AT 指令历史、统计数据等均必须保存在 SQLite 中。

代码应预留未来迁移至 MySQL 或 PostgreSQL 的能力，数据库访问层需保持良好的抽象和可扩展性。
